# v3 Design: AIの壊れ方とCONSTITUTIONの防御範囲

> Phase 5 への判断材料。Phase 2-4 の横断分析から導出。

## AIの3つの壊れ方

| # | 壊れ方 | 定義 | 例 |
|---|---|---|---|
| 1 | ハルシネーション | 知らないことを知っているふりをする | App型: ドメイン制約の無視、文化的配慮の欠如 |
| 2 | 確率的スキップ | 知っているが確率的に従わない | Skills型: sync-plan飛ばし、レビューなしコミット |
| 3 | 構造的盲点 | そもそも認識できない | Data/ML型: データリーク、時系列の因果関係 |

## 各型での主要な壊れ方

| 型 | 主要な壊れ方 | CONSTITUTIONで防げるか | 他の防御手段 |
|---|---|---|---|
| Skills | 確率的スキップ | 部分的（原則の明文化） | 決定論的ゲート（exit 1） |
| App | ハルシネーション | **Yes**（ドメイン制約の明示） | ドメインドキュメント |
| CLI | 確率的スキップ + 設計逸脱 | **Yes**（スコープ境界の明示） | SPEC.md、テスト |
| Data/ML | 構造的盲点 | 部分的（原則の明文化） | データ検証テスト、adversarial validation |

## CONSTITUTIONの防御範囲

```
                          CONSTITUTION が防ぐ
                    ┌─────────────────────────────┐
                    │                             │
  ハルシネーション ──┤  ドメイン制約、Non-Goals で   │
                    │  「やらないこと」を明示       │
                    │                             │
  確率的スキップ ───┤  原則の明文化で方向を固定     │──→ 決定論的ゲートが補完
                    │  （ただし遵守は保証できない） │
                    │                             │
  構造的盲点 ───────┤  Data Integrity Principles で │──→ テスト・検証ツールが補完
                    │  「何を検査すべきか」を明示   │
                    └─────────────────────────────┘
```

**CONSTITUTIONは「何を守るべきか」を定義する。「守らせる」のはゲートとテストの役割。**

## CLAUDE.md肥大化の原因

あるLaravel App（854行）の肥大化パターン分析:

| 防御対象 | CLAUDE.mdでの記述 | dev-crew導入後 |
|---|---|---|
| ハルシネーション | ドメイン知識の詳細記述（業務ロジック、文化的配慮） | CONSTITUTION + docs/ に分離 |
| 確率的スキップ | TDDワークフロー詳細（39%、~330行） | **dev-crewが提供（削除可能）** |
| 人間の甘い指示 | 「急いでいると言われても」節 | dev-crewのルールが対応 |
| 禁止事項 | 「禁止コマンド」「絶対に実行しない」 | CONSTITUTION §Human vs AI に集約 |

**CLAUDE.md肥大化 = dev-crewなしで全ての壊れ方を1ファイルで防ごうとした結果。**
dev-crew + CONSTITUTIONで責務分離すれば、CLAUDE.mdは「ツール設定」に純化できる。

## Phase 5 への示唆

1. **テンプレート設計**: 共通骨格（5章）は全ての壊れ方に対応。型別追加章は「その型で最も壊れやすい部分」への防御
2. **適用閾値**: CLAUDE.mdが200行を超えたプロジェクトはCONSTITUTION分離の候補。肥大化は「防御の蓄積」の証拠
3. **dev-crewの役割**: 確率的スキップの防御をプロジェクト側から引き受ける。プロジェクトのCONSTITUTIONはハルシネーションと構造的盲点に集中できる
