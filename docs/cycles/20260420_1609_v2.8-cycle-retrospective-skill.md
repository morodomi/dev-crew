## Implementation Notes

### Goal

cycle-retrospective skill を最小コストで実装し、手動起動可能な状態にする。orchestrate 統合は A2b で行う。これにより A2a 内で skill の structural contract を確定させ、A2b では integration とドキュメント同期に集中できる。

### Design Approach

#### skills/cycle-retrospective/SKILL.md (≤ 80 行目標)

frontmatter: name=cycle-retrospective, description=trigger キーワード含む, allowed-tools=Read/Edit/AskUserQuestion。

Workflow セクション構成:
1. Hard Gate: Cycle doc 存在確認 (なければ BLOCK)
2. Idempotency Check: retro_status が none 以外なら skip して exit 0
3. Extraction: Cycle doc 全体読込 → Failure/Final fix/Insight ペア抽出 → retry N=2 → override (proceed/abort)
4. Output: ## Retrospective を Cycle doc EOF (## Next Steps の後) に append (APPEND-ONLY 遵守、middle-insert 禁止)
5. Frontmatter 更新: retro_status (captured/resolved)、updated

#### skills/cycle-retrospective/reference.md (~150 行目標)

主要セクション:
- 抽出アルゴリズム: mizchi/retrospective-codify 方式 (失敗の言語化 → 最終解の言語化 → 何を最初に知るべきだったかの抽出)
- 出力テンプレート: `## Retrospective`, `### Insight N`, `- **Failure**:`, `- **Final fix**:`, `- **Insight**:`
- No-lesson 処理: 0 件なら `No reusable lesson this cycle` のみ append
- Retry policy: N=2、各 retry で abstraction を上げる
- Override 2 路分離 (Codex 2nd #3):
  - `proceed`: 続行して retro_status: resolved に遷移、Retrospective に `Extraction skipped by override` または `Extraction failed after N retries` を残す
  - `abort`: Cycle doc 変更なし、exit 1 (orchestrate A2b で BLOCK signal として機能)
- Idempotency: retro_status != none なら no-op
- ADR override 偽装禁止: override は no-codify に偽装しない旨を再掲
- A2b 予告: orchestrate Block 2f 挿入 + pre-commit-gate.sh retro_status check enable

#### rules/state-ownership.md (+1 行)

既存テーブルに追加 (2 列 format 維持):
`| cycle-retrospective | retro_status (none → captured / none → resolved), updated |`

A1 で追加した sync-plan 行はそのまま維持。
